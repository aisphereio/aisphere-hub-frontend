# aisphere-hub migration to aisphere-kit + aisphere-kit-kratos

This project was generated by `kratos new aisphere-hub` and then migrated to the
AI Sphere service starter model.

## What changed

1. `cmd/aisphere-hub/main.go` no longer manually loads Kratos config or calls Wire.
   It delegates lifecycle to `github.com/actionlab-ai/aisphere-kit-kratos/starter`.
2. `internal/app/app.go` is now the component composition root.
3. `configs/config.yaml` uses the shared `aisphere-kit/config.Config` schema.
4. `internal/data.Data` can hold `*starter.Runtime`, so repositories can use
   `rt.DB`, `rt.Redis`, `rt.S3`, `rt.Permission`, `rt.Authz`, `rt.Audit` later.
5. The generated Todo API is kept as a smoke-test service. Replace it with
   `api/hub/v1` Skill APIs in the next phase.

## Local development

If both kit repositories are private or still local, add replace directives:

```bash
go mod edit -replace github.com/actionlab-ai/aisphere-kit=../aisphere-kit
go mod edit -replace github.com/actionlab-ai/aisphere-kit-kratos=../aisphere-kit-kratos
go mod tidy
```

If the repositories are already pushed and tagged, use normal module resolution:

```bash
go get github.com/actionlab-ai/aisphere-kit@v0.1.0
go get github.com/actionlab-ai/aisphere-kit-kratos@v0.1.0
go mod tidy
```

## First run

The default config keeps DB/Redis/MinIO/Casdoor disabled so the skeleton can boot:

```bash
go run ./cmd/aisphere-hub
```

Then enable features one by one:

```yaml
features:
  db: true
  cache: true
  s3: true
  authn: true
  authz: true
  audit: true
  permission: true
```

## Next phase

Replace Todo with Hub domain APIs:

- `api/hub/v1/skill.proto`
- `internal/biz/skill.go`
- `internal/data/skill_repo.go`
- `internal/service/skill.go`

Permission changes should go through `rt.Permission`, which is backed by
Casdoor + Casbin. Do not create a local ACL table as the permission source.
